ProxSy

Proximity-based sharing for people on the same local area network. ProxSy combines Bluetooth Low Energy (BLE) for discovery (and optional direct BLE transfer) with TCP for fast, reliable payloads when both sides can reach each other over IP.

BLE discovery is useful when multicast DNS (mDNS/Bonjour) or similar LAN broadcast mechanisms are blocked or unreliable, because BLE advertisements still let nearby devices learn “who is here” and which IP and TCP port to dial.

What runs where

Layer Role
Wails desktop app (cmd/ui) Main UI: shows discovered peers, prepares messages/files, toggles transport and session modes, talks to Go backend.

Default TCP port in the UI entrypoint is 8090 (see cmd/ui/main.go).

Bluetooth Low Energy (BLE) — detailed behavior

Why two different BLE “stories”?

ProxSy uses BLE in two complementary ways:

  1. Discovery (always used for finding peers)
    Each device advertises a fake GATT primary service whose 128-bit UUID is not random — it is a bit-packed encoding of IPv4, TCP port, and a short device name. Nearby devices scan, parse matching UUIDs, and learn how to open a TCP socket.
    This avoids stuffing connection strings into the 31-byte legacy advertising payload; instead, the service UUID list in the advertisement carries the data in a form both macOS and Windows can set consistently.
  2. Optional “BLE-only” file/message transfer
    When the UI transport mode is BLE-only, payloads are sent over a separate fixed GATT service (12345678-1234-1234-1234-1234567890AB and associated characteristics — see drivers/mac_swift/Sources/main.swift and drivers/win_csharp/Program.cs). That path uses chunked writes, optional gzip, and SHA-256 verification — suitable when TCP is not desired or not available, at the cost of throughput.

Discovery UUID encoding (Mac and Windows must match)

The Swift advertiser (BLEAdvertiser.setupService) and the C# BuildProxsyUuid build the same string:

{hexIP}-{hexPort}-{seg1}-{seg2}-{seg3}
Segment Length Meaning
hexIP 8 hex chars (4 bytes) IPv4 address, each octet as two hex digits (e.g. c0a8012a → 192.168.1.42).
hexPort 4 hex chars (2 bytes) TCP port as big-endian unsigned 16-bit (e.g. 1f9a → port 8090).
seg1 + seg2 + seg3 4 + 4 + 12 hex chars = 20 hex chars = 10 bytes Byte 0 is magic 0xAB (ProxsyMagic). Bytes 1–9 are UTF-8 device name, zero-padded. Those 10 bytes are serialized as 20 hex characters, then split to satisfy the UUID 8-4-4-12 layout in the last three groups.

Scanner logic: a packet is treated as ProxSy if the reconstructed “name region” starts with 0xAB. The name is decoded from the following bytes up to the first 0x00. The scanner prints one CSV line per match to stdout:

{name},{ip},{port}

The Go adapters (pkg/discovery/ble/adapter_mac.go, adapter_windows.go) parse that line into ble.PeerDevice for the UI.

Advertisement fields (high level):

  • Service UUIDs: includes the encoded ProxSy UUID (this is how IP/port travel).
  • Local name: human-readable deviceName (also used for display and for matching in BLE-only transfer).

macOS nuance: BLEScanner may also connect to peripherals whose name contains "Prox" or "P|" to discover services if UUIDs were not present in the advertisement — a fallback for awkward stack behavior.

Advertise / scan duty cycle (pkg/discovery/ble/ble.go)

BLE_Initializer runs repeated cycles of concurrent advertise + scan, then optional rest, driven by a status string from the UI:

Mode (status) Advertise (s) Scan (s) Rest (s) Intent
active 30 30 0 Aggressive discovery.
default (passive) 10 10 20 Lower duty cycle / power saving.

Each cycle uses a child context so a mode change or timeout cancels the Swift/C# subprocesses promptly.
UpdatePeer / ReadPeers keep an in-memory peer list; CleanupPeers drops entries not seen for 60 seconds.

TCP path after BLE discovery

Once the client knows ip and port from BLE:

  1. It dials tcp://ip:port (pkg/client/client.go — RequestData).
  2. It sends a line: hostname|accessCode\n (access code used when session mode is enabled on the server).
  3. The server (pkg/server/server.go) may hold the connection in a pending approval pool until the user approves, then streams the prepared payload (text or framed file per pkg/converter).

So: BLE finds the socket; TCP moves the bulk data in the default mode.

BLE-only transport mode

When the UI sets transport to BLE-only (App.SetTransportMode in cmd/ui/backend/app.go):

  • Sending uses ble.FileSend / ble.MsgSend, which invoke the helper’s file_send / msg_send (and related) commands to push data over the fixed GATT transfer service to a peer matched by BLE local name.
  • Receiving runs ble.FileReceive in a loop, which calls file_receive on the helper; completed transfers are surfaced to the UI via Wails events.

Windows vs macOS CLI differences are encapsulated in the //go:build files: e.g. file_receive on Windows is invoked with (saveDir, timeout) while macOS includes deviceName as well — see adapter_windows.go vs adapter_mac.go.

Prerequisites

All platforms

  • Go (module declares go 1.24; use that or newer as required by your toolchain).
  • Node.js 18+ for the UI frontend.
  • Wails v2 CLI, e.g. go install github.com/wailsapp/wails/v2/cmd/wails@latest.

macOS

  • Xcode Command Line Tools (for swiftc and codesigning in build.sh).
  • Python 3.11 if you run build.sh (it sets up a venv under hand-gesture-recognition-mediapipe-main/myenv for MediaPipe-related steps).

Windows

  • .NET 8 SDK to publish ble_windows.exe.
  • npm for cmd/ui/frontend.

Building

macOS (build.sh)

The script:

  1. Creates/updates a Python 3.11 venv and installs MediaPipe-related dependencies (for the bundled hand-gesture project).
  2. Compiles a universal execs/ble_mac from Swift (arm64 + x86_64 with lipo).
  3. Runs wails build in cmd/ui.
  4. Copies the app to /Applications/ProxSy.app, embeds ble_mac inside the bundle, applies entitlements from drivers/mac_swift/Sources/entitlements.plist, and creates a proxsy_run symlink at the repo root.
chmod +x build.sh
./build.sh

Windows (build.ps1)

The script:

  1. Runs npm install and npm run build in cmd/ui/frontend (required for embedded frontend/dist).
  2. dotnet publish the C# project to execs/ as ble_windows.exe (single-file style publish).
  3. wails build in cmd/ui, then moves win_proxsy.exe to execs/.
.\build.ps1

Running during development

From cmd/ui (with Wails and frontend deps installed):

cd cmd/ui
export PATH="$HOME/go/bin:$PATH" # if wails is installed via go install
wails dev

Ensure execs/ble_mac or execs/ble_windows.exe exists from a prior driver build, or set an explicit path:

Variable Purpose
PROXSY_BLE_EXE Full path to the BLE helper if it is not under execs/.
PROXSY_REPO_ROOT Absolute repo root when the app runs from a temp directory and cannot walk up to find execs/.

pkg/binarypath/binarypath.go documents the search order.

Finding the BLE helper from the UI

The backend exposes GetBLEDriverPath() so the UI can show whether the driver was resolved (empty string means no helper found — BLE features will not work).

Windows troubleshooting (BLE helper)

  • Unblock downloaded or copied .exe files: Properties → Unblock.
  • Developer Mode can be required for some WinRT Bluetooth scenarios (Settings → Privacy & security → For developers).
  • Build for the right runtime: scripts use win-x64 (dotnet publish … -r win-x64).

Historical note (older README)

Earlier documentation described 6-byte manufacturer-specific data for discovery. The current implementation uses the custom 128-bit service UUID scheme above so Mac (CoreBluetooth) and Windows (GattServiceProvider) stay in sync. If you see references to “6-byte manufacturer data” elsewhere, treat them as outdated unless the code is changed to match.
